“On a payments engagement, how do I handle duplicate transaction requests?”

The mechanism is an idempotency key — but the requirement nobody volunteers is the duplicate-and-retry path.1

Stakeholders describe the happy path and leave “what happens if the same request arrives twice” unspoken. Surface it. In the FRD, specify that requests carrying an identical key within the dedupe window constitute a single logical operation, and add an acceptance criterion: a duplicate key replays the original response with no second debit.2

The design decision this forces is where the key lives and how long the window holds — a longer window is safer but carries more state.3 For a Canadian D-SIB, it anchors to LYNX exactly-once expectations and OSFI E-21 operational continuity.4
